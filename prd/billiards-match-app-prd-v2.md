## 1. 二期核心目标

一期解决了"一张桌子边的几个人，拿起手机就能计分"。二期要跨越两个边界：

| 边界 | 问题 | 二期解法 |
|------|------|----------|
| **跨设备** | 多人想同时看比分、或记分者换人 | 共享比赛：扫码/分享加入，实时同步 |
| **跨场景** | 球房想办赛事，球友想找比赛 | 球房入驻 + 赛事发布 + 在线报名 |

实现这两件事，**必须引入服务端**。这是二期与一期最本质的差别。

## 2. 范围 & 非目标

### 2.1 二期做（MVP）

1. 账号系统（微信小程序 + 抖音小程序快捷登录 + 手机号登录；游客模式降级保留；跨平台同账号合并）
2. 共享比赛（创建 → 分享 → 加入 → 实时同步 → 结束保存）
3. 球房入驻（申请 → 审核 → 主页展示）
4. 赛事发布 & 报名（球房发布 → 球员浏览 → 在线报名 → 赛程生成 → 现场计分 → 结果公示）
5. 与一期的互操作：原有本地记分路径保留；登录后本地记录可选择同步到云端

### 2.2 二期**不做**（留给三期+）

- 球友圈（好友/动态/约球/附近球友）——独立模块，工作量巨大
- 球杆商城（支付、物流、售后）
- 奖金托管/发放（二期只做信息展示）
- 排位系统 / ELO 积分
- 付费赛事报名（二期仅免费）
- 视频直播 / 裁判辅助
- 海外支付 / 国际化

不做这些是为了聚焦两件主事，并避免合规复杂度（支付、实名、版权）。

## 4. 场景一：共享比赛

> **定位**：球房里几个人一起玩，手机分一下便于共同记录。**不是一场正式赛事**，目标是把时间留给打球娱乐，App 做减法。

### 4.1 场景叙事

> 张三、李四、王五在球房打九球。张三创建了一场比赛，把玩家名填进去后点"分享"生成小程序码。李四扫码进入后点了 2 号位；王五扫码点了 3 号位——三个号位全满。朋友老六也扫了码围观，他没占位，就是观众。开打后三人谁拿到手机都能点记分；老六只能看。第三局张三手滑把"普胜"点成了"大金"，李四发现后点"撤销"，状态回到上一步。两小时后张三点 ✕ 结束，三人的历史记录里都有这场。

### 4.2 两个核心概念（极简）

只保留两种身份，取消单独的"记分员"角色：

| 身份 | 判定 | 能做什么 |
|------|------|----------|
| **参赛者** | 占了某个号位的登录用户 | 记分、犯规、暂停/继续、改名、撤销、分享、结束比赛 |
| **观众** | 没占号位的用户（含未登录游客） | 只读：看比分、看操作日志 |

**房主**不是单独身份，只是"创建者"这个事实。除了"结束比赛"和"开赛前改规则"这两件事默认只有房主能做之外，其他所有参赛者权利平等——**任何占位者都能记分、都能撤销**。

**为什么放开**：球房里都是朋友，权限复杂带来的摩擦远大于误操作风险；所有操作有完整日志 + 任何人可撤销，误操作可以补救。

### 4.3 房间码 & 邀请

| 概念 | 说明 |
|------|------|
| 房间码 | 6 位大写字母+数字（如 `K7P2XM`）；比赛存活期间有效 |
| 邀请链接 | 微信分享卡 / 小程序码；落地页 `/pages/join/index?code=XXXXXX` |
| 码的生命周期 | 比赛"进行中/暂停"有效；"结束"后立即失效 |

### 4.4 加入流程

```
扫码 / 打开链接
    │
    ├─ 未登录
    │   ├─ 有人等你占位？→ 提示登录
    │   └─ 围观？→ 直接进观众视图
    │
    └─ 已登录
        │
        ├─ 号位有空 → 弹出"选择你的号位"→ 点选即占位（变参赛者）
        ├─ 号位已满 → 直接进观众视图
        └─ 你就是这场的参赛者（重连）→ 自动恢复号位
```

**交互细节：**

- 参赛者随时可"离位"变观众（号位空出给别人）
- 观众随时可"占空位"变参赛者
- 任何参赛者可"让座"：把自己的号位让给场内观众（一键）

### 4.5 操作日志 & 撤销

**核心机制：所有改变比赛状态的操作都是"事件"**，持久化、可见、可撤销。

**日志条目（MatchEvent）：**

| 字段 | 说明 |
|------|------|
| `id` | 事件 id |
| `matchId` | 所属比赛 |
| `actorUserId` | 操作人（参赛者才会生成事件） |
| `type` | `score_win` / `foul` / `rename` / `pause` / `resume` / `undo` / `end` / `seat_change` |
| `payload` | 语义化参数（哪个玩家普胜、掏了谁、等等） |
| `serverSeq` | 服务端递增序号（权威顺序） |
| `undone` | 是否已被撤销（bool） |
| `undoneBy` | 撤销此事件的 eventId |
| `createdAt` | 时间戳 |

**撤销规则：**

- 任何参赛者可点"撤销" → 撤销**最近一条未被撤销的普通事件**（不含 `undo` / `end`）
- 连续点可以撤销多步（按 `serverSeq` 倒序）
- **撤销本身也是事件**，记入日志但不可被撤销（避免 undo loop）
- 被撤销事件的 `undone=true`，状态机在计算最终状态时跳过它
- 撤销后广播给所有端，UI 实时刷新

**操作日志 UI：**

- 比赛页顶栏多一个 🕘 图标 → 弹出"操作记录"抽屉
- 时间倒序，每条显示：操作人头像 + "14:32 张三 记了李四一个普胜"
- 每条末尾显示"已撤销"（灰色）或留白
- 抽屉顶部一个大按钮"撤销最近一步"（参赛者可点）

### 4.6 并发冲突

多人同时操作时：

1. **事件模型**：客户端发"张三普胜掏李四"这类语义事件，不发 `{张三: 32}`
2. **服务端串行处理**：按到达顺序赋予 `serverSeq`，依次应用到状态机
3. **乐观 UI**：本地立即应用，同时发送事件；服务端回包若 `serverSeq` 连续则确认，否则重放补齐
4. **最终一致**：服务端是权威，所有客户端最终收敛
5. **不做操作防抖**：反正能撤销

### 4.7 断网 / 重连

| 情况 | 行为 |
|------|------|
| 参赛者短暂掉线（<60s） | UI 提示"重连中"，本地操作队列暂存 |
| 参赛者长时间掉线 | 号位保留（不自动放空）；重连后继续 |
| 房主掉线 | 房间不因此终止，其他参赛者照常玩 |
| 所有人掉线 | 15 分钟后服务端自动"暂停"；2 小时后自动"结束"并持久化 |
| 重连 | 客户端带 `lastServerSeq` 请求增量；服务端补发 |

## 5. 场景二：球房入驻 + 赛事

> **定位**：跟共享比赛是不同物种。共享比赛 = 朋友局 2-3 人；赛事 = 16 / 32 / 48 / 64 / 128 人，可能跨数日，必须有完整的赛制和签表逻辑。

### 5.1 场景叙事

> 老赵的"黑八桌球会所"刚通过平台入驻审核。准备办"夏日挑战赛" —— 预计 32 人，他配置：项目中八、**预选小组赛**（4 人/组，每组出 2 名）→ **正赛单败淘汰**（16 人），周六周日两天打完。
>
> 报名开放后两天报满 32 人。周六上午 10 点开赛，系统按"蛇形"把 32 人分 8 组每组 4 人，每组 6 场循环，每场打完通过共享比赛能力记分。所有小组赛打完，系统自动按战绩积分排出每组前 2 共 16 人，进入正赛 bracket。
>
> 周日下午正赛打到决赛。老赵这次还选了**双败** —— 输一场不直接出局，进败者组还有翻身机会。最后败者组冠军和胜者组冠军打总决赛，结果出来后系统自动公示冠亚季军，所有参赛者历史里都关联了这次赛事。

### 5.2 角色分工

- **球房管理员**：创建球房 → 运营主页 → 发布赛事 → 现场主持
- **球员**：发现球房 → 浏览赛事 → 报名 → 到场参赛
- **平台运营（后台）**：审核入驻申请

| 营业时间 | 选填 | 文字 |

### 5.4 赛制模型 ⭐ 核心

赛事的赛制不是单一类型，而是**一个或多个"阶段（Phase）"按顺序执行**，每个阶段输入若干参赛者，输出晋级者交给下一阶段。

#### 5.4.1 阶段类型（四种）

| 类型 | 说明 | 出线规则 |
|------|------|----------|
| **小组赛**（GroupStage） | N 人分若干组，每组内循环对战 | 按胜场/积分排名，每组取前 K 名 |
| **单败淘汰**（SingleElim） | 标准 bracket，输一场出局 | 决出冠军 |
| **双败淘汰**（DoubleElim） | 胜者组 + 败者组双 bracket，败者组冠军和胜者组冠军打总决赛 | 决出冠军 |
| **循环赛**（RoundRobin） | 所有人互相打一遍 | 按战绩排名 |

#### 5.4.2 推荐赛制模板（MVP 提供一键选）

球房常用的不是任意组合，是几个固定模式。MVP 只开放以下模板：

| 模板 | 阶段构成 | 适用规模 | 时长估计 |
|------|----------|----------|----------|
| **快速单败** | SingleElim | 4–32 人 | 半天 |
| **快速双败** | DoubleElim | 4–16 人 | 一天 |
| **小组赛 + 单败正赛** | GroupStage → SingleElim | 16–128 人 | 1–2 天 |
| **小组赛 + 双败正赛** | GroupStage → DoubleElim | 16–64 人 | 2 天 |
| **纯循环** | RoundRobin | 4–8 人 | 半天 |

未来 v2.1+ 可开放"自定义阶段链"，MVP 不做。

#### 5.4.3 各阶段配置项

**SingleElim（单败）：**

| 字段 | 说明 |
|------|------|
| 参赛人数 | 上一阶段晋级人数 / 或直接报名人数 |
| 种子规则 | 随机 / 上阶段排名 / 报名顺序 / 手动 |
| 三四名决赛 | 是否打（默认是） |

**DoubleElim（双败）：**

| 字段 | 说明 |
|------|------|
| 参赛人数 | 同上 |
| 种子规则 | 同上 |
| 总决赛规则 | "败者组胜者赢一次即夺冠" / "必须连赢两场"（对冲败者组已输过一次） |

**GroupStage（小组赛）：**

| 字段 | 说明 |
|------|------|
| 参赛总人数 | 来自报名 |
| 每组人数 | 3 / 4 / 5 / 6（推荐 4） |
| 每组出线数 | 1 / 2 / 3（推荐 2） |
| 排名规则 | 胜场 → 总分差 → 互胜（中八）；胜场 → 净胜分（九球） |

**RoundRobin（循环）：** 和小组赛类似，但不分组，全体互打一遍。

#### 5.4.4 非 2 的幂人数处理

实际报名常是 27 / 48 / 100 这类非 2 幂：

1. **Bracket 阶段加 BYE（轮空）**：把人数补到下一个 2 幂；例如 27 → 32 节点，5 个高种子首轮轮空晋级
2. **小组赛不需 2 幂**：48 人分 12 组每组 4 人出 2 → 24 人进正赛，再加 8 个 BYE = 32 bracket

**配置时系统自动校验 + 预览**：填完后展示"共 X 阶段、Y 轮、Z 场，估计 H 小时"；数字不合理（晋级人数无法凑成合理正赛）→ 报错 + 给修改建议。

### 5.5 赛事生命周期

```
[草稿] --publish--> [报名中] --报名截止--> [待开赛]
                                             │
                                             ├--到开始时间--> [进行中]
                                             │                    │
                                             │           Phase 1 ─┤
                                             │     (本轮所有 Match 完→下一轮)
                                             │     (本阶段所有轮完→下一阶段)
                                             │           Phase 2 ─┤
                                             │                    │
                                             │                    ↓
                                             │                 [已结束]
                                             │
                                             └--cancel--> [已取消]
```

赛事可能跨数日。**轮次推进不依赖管理员手动操作** —— 当前轮所有 Match 打完，下一轮自动生成；本阶段所有轮完成，自动进下一阶段。管理员可发公告（"下一轮今晚 7 点开始"），但不堵流程。

### 5.6 对阵表自动生成

**SingleElim（标准 bracket seeding）**：

```text
8 人为例（种子 1-8）：
首轮：(1 vs 8)、(4 vs 5)、(3 vs 6)、(2 vs 7)
半决：胜者(1/8) vs 胜者(4/5)；胜者(3/6) vs 胜者(2/7)
决赛：两个半决胜者
```

**DoubleElim**：胜者组失败者按标准算法流入败者组对应位置，最后败者组冠军 vs 胜者组冠军打总决赛。通用算法，实装时引用库或自实现。

**蛇形分组（GroupStage）**：种子按蛇形均匀分布，保证每组难度均衡。

```text
12 人 4 组：
Group A: 1, 8, 9
Group B: 2, 7, 10
Group C: 3, 6, 11
Group D: 4, 5, 12
```

**自动晋级**：一场 Match 结束（matchEnded 事件）→ 服务端找到关联 BracketNode → 写 winner/loser → 若本轮所有节点都有结果 → 自动生成下一轮 → 若本阶段所有轮完成 → 启动下一阶段 → 若最后阶段完成 → 赛事结束，公示冠亚季军。

### 5.7 赛事发布流程（球房管理员）

#### 5.7.1 基础字段

| 字段 | 说明 |
|------|------|
| 标题 | "夏日挑战赛" |
| 球房 | 自动 = 管理员名下的球房 |
| 项目 | 中八 / 九球 |
| 报名上限 | 任意正整数（4 起；MVP 建议 ≤128） |
| 赛事开始时间 | 日期时间 |
| 报名截止 | 默认 = 开始前 1 小时 |
| 费用说明 | 文本（二期不走支付） |
| 奖品说明 | 文本 |
| 补充说明 | 文本 |

#### 5.7.2 赛制配置（两步 + 预览）

1. **选模板**（§5.4.2 的 5 个）
2. **补参数**（每阶段按 §5.4.3）
3. **预览页**：系统计算并展示
   - 共需 X 阶段；每阶段 Y 轮；每轮 Z 场
   - 估计总场次 N；按 30 分钟/场估计总时长
4. 确认 → 创建赛事（草稿态）→ 继续编辑 → 发布

#### 5.7.3 单场规则

每场比赛内部规则由赛事统一注入：

- **九球赛事**：必须配置目标分（默认 51），得分规则可改
- **中八赛事**：必须配置抢几局

（注意：一期消遣模式"九球无目标分"，但**赛事场必须有终止条件**，所以赛事里的九球强制配目标分。）

```text
小李
──
① "发现" → 搜索球房 / 浏览附近赛事
② 进入赛事详情页
③ 点"报名"

```

**取消报名**：截止前自主；截止后联系球房管理员。

### 5.9 赛事进行流程

开始时间到 → 自动启动（或管理员点"开始"）：

1. 启动**第一阶段**，按种子规则生成对阵
2. 每场对局自动 `createMatch`，关联 `eventId` + `phaseIndex` + `bracketNodeId`
3. 对局双方在赛事页看到自己的对阵 + **进入比赛**按钮，点击直达共享比赛
4. 计分走 §4 流程；终止由赛事规则决定（中八抢局数 / 九球目标分）
5. matchEnded → 服务端写 winner/loser 到 BracketNode → 判断本轮/本阶段是否完成 → 自动推进

**球房管理员可介入：**

- 标记某人弃权（计为一场败）
- 改桌号、发公告下一轮时间
- 强制结束某场（判罚）
- 修正比赛结果（异常情况；写审计日志）

**服务端技术栈**（供讨论，未选型）：

- 后端语言：**Node.js (NestJS) / Go / Python(FastAPI)** 任选一；倾向 Node（与前端同栈，团队转切成本低）
- 数据库：MySQL 或 PostgreSQL
- 实时：Redis Pub/Sub + WebSocket（或直接用微信小程序的 WebSocket 能力）
- 云：腾讯云 / 阿里云（在国内离微信生态近）
- **备选极简路径**：直接用**微信云开发** —— 数据库/函数/存储一套搞定，省运维但锁定生态

### 6.3 实时同步策略

- **主通道**：WebSocket，客户端和服务端之间长连
- **事件模型**：客户端发 `{type, matchId, payload, clientSeq}`；服务端处理后广播 `{type, matchId, state, serverSeq}`
- **重连**：客户端维护 `lastServerSeq`，重连后拉增量
- **降级**：WS 不可用时退化到 HTTP 长轮询（每 3s 拉最新状态）

### 6.4 账号系统

一期 PRD（§4.1 客户端矩阵）明确微信小程序 P0 + 抖音小程序 P1，所以 **C 端账号要同时支持微信和抖音登录**，并能把两个平台关联到同一个"人"。

**登录路径：**

- **微信小程序**：`wx.login` → `code2session` → 服务端换 openId / unionId → 返回统一 token
- **抖音小程序**：`tt.login` → 服务端 code2session（抖音开放平台） → 返回统一 token
- **手机号登录**（备）：短信验证码，兜底 H5 场景；绑定手机号后成为**主识别键**
- **游客**：保留一期能力；未登录时只能本地计分 + 观战共享比赛
- **Token**：JWT（short-lived access + refresh），服务端可吊销

**账号关联 / 合并规则：**

- 同一手机号 → 同一 User
- 同一微信 unionId → 同一 User
- 同一抖音 unionId → 同一 User
- **跨平台合并仅在双方都绑定同一手机号时触发**——避免误合
- 用户可在"我"页主动绑定手机号把两个小程序的账号合并
- 运营也可在管理后台手动 merge（`POST /admin/users/merge`，仅 super_admin，写审计）

**数据模型：** 见管理后台设计稿 §4.6 的 `User / WechatBinding / DouyinBinding`。

## 9. 待确认问题（请你过一遍）

### 9.1 产品范围

1. **游客**能否在"发现"里看到球房/赛事信息（只读）？还是必须登录才能进发现页？
2. 球房入驻**是否要人工审核**？如果平台运营还没配备，是否先做"自助+事后抽查"？
3. 赛事报名**二期一定免费吗**？如果想收费，会大幅增加复杂度（支付合规）——建议二期坚决免费
4. 赛制**只做单败淘汰**够不够？你经常听到球房办哪种赛制？
5. 一期本地记录**要不要**一键导入云端？还是让新账号重新开始？

### 9.2 共享比赛细节

6. **"观众也能参与记分"** 这个功能要不要？目前设计只有房主和授权记分员能改分，观众纯旁观。如果观众也能改，可能出现"群攻"改分的混乱
7. **旁观者身份要不要登录**？不登录的话怎么识别是哪个路人？目前方案：不登录=临时观众（无个人记录），登录=可被授权升级
8. 比赛**中途换人**（李四不打了，王五上场 2 号位）要支持吗？会让数据模型复杂（分数归谁）
9. 房主**能不能转让房主身份**给其他人（中途走人）？

### 9.3 球房 & 赛事

10. 球房管理员是否必须是**球房老板本人**？还是老板可以指定店员做管理员？（关系到账号是否可多人共享）
11. 赛事**结果上传**要不要允许管理员手动覆盖？比如系统自动判了但实际有争议
12. 赛事**奖金信息**纯文本展示（"冠军免 3 小时台费"）够不够？还是要结构化（金额、等级）？
13. **付费台费**场景有没有：球房想在 app 里收小时费？（我的建议：二期不做，三期再议）

### 9.4 技术路线

14. 服务端**栈偏好**：Node.js / Go / Python / 微信云开发？（团队技能决定，我没立场）
15. **微信云开发**作为 MVP 加速器怎么样？缺点是深度绑定微信生态，跨端受限
16. 是否有**预算/部署偏好**（腾讯云、阿里云、自建机房）？

### 9.5 发布策略

17. **先做共享比赛还是先做球房赛事**？我建议共享比赛先——它能独立闭环，且为赛事流程打基础
18. **内测渠道**：球房渠道获客策略（一期 PRD 提到）还保留吗？
