# 桌球计分 · v2.21 「附近」改按城市 PRD

> **版本：** v2.21.0 草稿
> **分支：** `v2.21`（基于当前 `v2.20`）
> **更新时间：** 2026-05-28
> **状态：** 设计中（等用户 review）
> **依赖：** v2.10 球房/赛事已落地，已申请 `scope.userFuzzyLocation` 权限

---

## 0. TL;DR

- 把首页「发现球房 · 附近球房」从**假标签**变成**真按城市**：用户当前所在城市 → 该城市的球房列表，跨城自动给提示。
- 报名赛事时，若赛事所在城市 ≠ 用户当前城市，**弹二次确认**：「该赛事在 xxx 市举办，距离较远，确认报名吗？」
- 新增 `Venue.city` 字段；新建/审核通过的球房自动从 `address` 解析出 `city + lat + lng`。
- 客户端定位走 `wx.getFuzzyLocation`（已申请），逆地址解析放服务端，封一个 `/v1/geo/resolve-city` 接口；外部依赖 = 腾讯地图 WebService。
- 用户拒绝授权 → 降级到「手动选城市」；首次定位结果本地缓存 24h，可以手动改。

## 2. 用户故事

1. **小张在济南**：打开小程序首次进入「发现球房」→ 弹微信授权框「允许使用你的位置」→ 同意后看到的是「济南」标签 + 济南的球房；不同意降级到「请选择城市」。
2. **小李刷到一个青岛的赛事**：点报名 → 弹「该赛事在**青岛市**举办，距离较远，确认报名吗？」→ 二次确认才进报名流程。
3. **小王临时出差到北京**：在「发现球房」页顶部点城市标签「济南 ▾」→ 弹城市选择器 → 切到「北京」→ 列表刷新；选择本地缓存 24h。

## 3. 设计要点

### 3.1 客户端定位流程

```
进入 venues/tournaments 页
  → 读本地 storage `userCity`（24h 内有效）
    ├─ 有 → 直接用
    └─ 无 → wx.getFuzzyLocation
              ├─ 成功 → POST /v1/geo/resolve-city → 拿到 city → 写 storage
              ├─ 用户拒绝授权 → 降级：弹 toast「未授权，请选城市」+ 自动打开城市选择器
              └─ 接口失败/超时 → 降级同上
顶部一直显示当前城市，可点击切换
```

### 3.2 服务端逆地址解析

- 新接口：`POST /v1/geo/resolve-city` body `{ lat, lng }` → `{ city: "济南市", province: "山东省" }`
- 实现：调腾讯地图 WebService [`/ws/geocoder/v1`](https://lbs.qq.com/service/webService/webServiceGuide/webServiceGeocoder)（逆地址解析）
- key 配在服务端环境变量 `TENCENT_MAP_KEY`，**绝不**下发到前端
- **缓存**：lat/lng 按 0.01° 网格（约 1km）取整作为 key，存 Redis/内存，TTL 30 天；同一格子的请求共用结果。控制配额。

### 3.3 球房入库时反向写 city + lat/lng

- 商家审核通过 / 后台直接建房 / 商家更新地址 时，触发地理编码
- 调腾讯地图 [`/ws/geocoder/v1`](https://lbs.qq.com/service/webService/webServiceGuide/webServiceGeocoder)（正向地址 → 经纬度 + 行政区划）
- 写入 `Venue.city / lat / lng`
- 失败兜底：保留 `city = null`，列表筛城市时不出现；admin 后台允许手填修复

### 3.4 列表 / 筛选 / 排序

- `GET /v1/venues?city=济南市` → 优先按城市筛
- 同城内按 `(lat,lng)` 距离用户位置升序（如果客户端传了用户经纬度），否则按 `createdAt` 降序
- 客户端调用：进入页面默认带 `city = userCity` 和 `lat/lng`

### 3.5 跨城报名二次确认

- 入口：`pages/tournament-detail/index` 点「立即报名」按钮
- 取赛事 venue 的 `city` 与本地 `userCity` 对比
- 不一致 → `Taro.showModal({ title: '跨城报名确认', content: \`该赛事在 \${tournamentCity} 举办，距离较远，确认报名吗？\` })`
- `userCity` 为空（用户没授权也没手选）→ 不弹（避免骚扰）；可选：弹「请先在 发现球房 页选择当前城市」

### 3.6 城市选择器

- 数据源：MVP 阶段硬编码省 → 市两级菜单（覆盖直辖市 + 省会 + 主要地级市，约 100 个）
- 复用：`venues` 页和 `tournaments` 页顶部都挂城市切换入口
- 后续：等积累商家足够多了再改成「按 venue.city 聚合的可选城市」

## 4. 接口清单

### 4.1 新增

| 方法 | 路径 | 鉴权 | 说明 |
|---|---|---|---|
| POST | `/v1/geo/resolve-city` | 无（带速率限制） | body `{lat, lng}` → `{city, province}` |

### 4.2 改动

| 方法 | 路径 | 改动 |
|---|---|---|
| GET | `/v1/venues` | 加 query：`city`（精确匹配）、`lat`、`lng`（按距离排序） |
| 列表响应 `VenuePublic` | | 新增 `city: string \| null` |
| 列表响应 `TournamentDetailPublic.venue` | | 新增 `city: string \| null` |

### 4.3 商家侧（admin-web）

- `PATCH /v1/venue/me/venue`：address 改动时触发服务端地理编码，覆盖 `city / lat / lng`
- 后台审核 `POST /v1/admin/venue-applications/:id/approve`：审核通过创建 venue 时同步写 city
- 后台 venue 编辑页：city 字段可读可写（兜底人工修复）

## 5. 数据库变更

```prisma
model Venue {
  // 既有字段...
  city  String?  @db.VarChar(32)
  // lat / lng 已存在

  @@index([city])
}
```

**Migration**：

1. `prisma migrate dev --name add_venue_city`
2. 一次性脚本 `scripts/backfill-venue-city.ts`：遍历 city == null 的 venue，调腾讯地图地理编码 API 回填 city/lat/lng；失败的打日志、留 null。

## 6. 隐私与合规

- `app.json` 加：
  ```json
  "requiredPrivateInfos": ["getFuzzyLocation"]
  ```
- 「我」页 → 隐私协议 / 用户协议正文，补一段：「我们获取你的模糊位置，仅用于展示附近球房与赛事，不用于其他用途，不与第三方共享。」
- 用户拒绝 / 撤回授权 → 不影响其他功能使用，仅城市相关功能降级到手动选。
- 服务端**不持久化**用户经纬度，仅一次性用于换城市；日志里也只记 city，不记 lat/lng。

## 7. 外部依赖与配额

- **腾讯地图 WebService key** 1 个，需注册「位置服务」开发者账号
- 配额：免费 1 万次/日（账号实名）；逆地址 + 地理编码合算
- 以网格缓存 + 用户 24h 缓存压配额，估算正常使用每日 < 1000 次
- 失败重试：地理编码失败重试 1 次，仍失败写 null + 日志

## 9. 风险与已知限制

- **微信侧**：`getFuzzyLocation` 要求基础库 ≥ 2.25.0；低版本走降级到手选城市。
- **腾讯地图**：免费额度耗尽 → 接口直接返回 `{city: null}`，前端走手选降级（不阻塞用户）。
- **回填脚本**：旧 venue 地址不规范（如「XX路」无市级前缀）解析率不会 100%，admin 后台允许手填补救。
- **iOS 用户**：首次弹位置授权框时间不可控，授权框不能完全自定义文案；在弹之前我们先 toast 一行说明「即将申请位置权限用于发现附近球房」。
- **跨城判断口径**：以「市」为单位（济南市 vs 青岛市）。MVP 不做「同省不同市」的特殊降级（省内可能也很远）。
