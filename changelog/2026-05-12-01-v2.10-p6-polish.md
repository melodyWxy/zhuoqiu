# v2.10 P6：OSS 直传 + 联机记分体验打磨 + 赛事开赛 bugfix

v2.10 收尾：把试跑里发现的几类问题一次性收掉，准备切 v2.20。改动按"基础设施 / C 端体验 / 赛事链路"三块组织，单次提交。

## 一、商家上传切阿里云 OSS（STS 直传）

P1 的本地 multipart 上传在 server 带宽和文件持久化上有明显短板（生产 server 卷不应该当文件存储用）。改成 STS 直传：

### 服务端

- 新增 `OssStsService`（`server/src/upload/oss-sts.service.ts`，191 行）
  - `AssumeRole` 拿 900s 临时凭证；Policy 限 `oss:PutObject` + `{category}/{yyyymmdd}/*` 前缀
  - category 白名单：`license / id-card / venue-cover / tournament-cover / avatar / general`
  - 提供 `isValidObjectUrl` 给后续业务接口校验前端回传 url 是否指向本 bucket
- `UploadController`：
  - 新增 `GET /uploads/sts-token?category=xxx`（@VenueAuthGuard）
  - `OSS_ENABLED=true` 时 `POST /uploads` 主动拒绝（强制走直传）
  - `OSS_ENABLED=false` 兜底回退本地 `uploads/` 卷，便于 dev 离线跑
- `.env.example` 新增 OSS 配置块（默认 `OSS_ENABLED=false`，dev 不受影响）
- 依赖：`@alicloud/pop-core` + `ali-oss`

### 三端客户端

- `admin-web/src/api/venue.ts` `uploadApi.upload`：multipart → ali-oss SDK `client.put`
- `billiards-score/src/core/api/venue.ts` `uploadVenueFile`：H5 端同上；小程序端先抛错（待接 uni-upload）
- 两端都做 STS token 缓存：剩余 >60s 直接复用，避免每次都 `AssumeRole`

## 二、C 端联机记分页体验打磨

P5 跑完闭环后跑了一场 4 人赛事真机 demo，反馈集中在三个点：

### 1. 操作历史抽屉（追溯能力）

- 新增 `components/MatchHistorySheet`：弹出抽屉，按 serverSeq 倒序列所有事件
  - 翻译事件类型：`score_normal_win` → `张三 普胜（掏 李四）`、`foul` → `张三 犯规，李四 +1` 等
  - 每行带 actor：`管理员 王五` / `张三（138****1234）` / `用户 abc12345…` / `系统`
  - 已撤销操作灰显 + "已撤销"标
- 服务端 `MatchService.getEvents` 配套：`serverSeq desc` + 关联 `author` 取 nickname/phone（脱敏后再返回），管理员操作单独 `adminAccount.findMany` 拿 name/username
- 联机页顶栏新增"📜 历史"按钮触发

### 2. 连击保持选中 + +N toast

- 之前每次记分后清空 `selectedSlot`，连续给同一人加分要再选一次玩家
- 改成：成功后保持选中，弹 `+N` toast（N 来自 rules.normalWin/bigJack/...，golden9 默认 4）
- 顺手修了 Taro H5 下 `onClick` 闭包读到旧 `selectedSlot` 的 bug：加 `selectedSlotRef`，`handleWin/doWin` 走 ref.current

### 3. 操作按钮统一为 ops-grid

- Taro `<Button>` 原生样式各端不一致（H5 / 微信 / 抖音差异显著）
- 全替换为自渲染 `<View className="op-btn">`，3×2（九球） / 2×1（八球）grid，等高 52/56px
- 颜色按动作类型分组：win-* / op-foul / op-undo

### 4. 重连兜底（第一次见）

之前 `MatchSocket.scheduleReconnect` 是无穷次拨号，断网 30 分钟后回桌面会看到 console 持续 print。

- `MAX_RECONNECT=8`（退避到 15s 上限后约 2 分钟封顶）
- 监听 `visibilitychange / online`：标签切回 / 网络恢复立即拨一次（不等退避）
- 认证类关闭码（4001/4003/1008）直接放弃，不再重连
- 抛 `__ws_open__ / __ws_close__ / __ws_gave_up__` 内部事件给 UI
- 新增 `components/ConnectionBanner`：黄/绿/红横条 + "重新连接 / 刷新页面"

### 5. 杂项

- `pages/config/index.tsx`：登录后默认昵称用 `cloudUser.nickname`，赛事 / 联机昵称同源
- `pages/join/index.tsx`：换 `PageHeader` 与其他页对齐

## 三、赛事开赛链路 bugfix

P5 真机 demo 暴露的几个问题：

### 1. rules JSON 缺字段导致 `NaN` → 序列化成 `null`

赛事的 `tournament.rulesJson` 通常只装 `raceToWins`，没存 `normalWin/bigJack/golden9` 这些记分细则。bracket 开 match 时直接把它当 match.rulesJson 落库，state-machine 跑分时 `score + undefined = NaN`，然后 `JSON.stringify` 成 `null`，前端拿到一个 `null` 分数无法渲染。

- `server/src/match/state-machine/nine-ball.ts`：`normalizeRulesRuntime` 把 undefined 字段回落到 `DEFAULT_NINE_BALL_RULES`
- `server/src/venue/bracket.service.ts`：`mergeRulesForMatch` 创 Match 时合并默认值落库（双保险，老数据也能跑）

### 2. 商家 JWT 里 `venueId` 滞后

老板第一次申请通过后，DB 里 `VenueAccount.venueId` 已经更新，但他登录时拿到的 JWT 还是 `venueId=null`。下次他点"创建赛事"会报 `VENUE_NOT_FOUND`。

- `TournamentMerchantController.resolveVenueId`：JWT 没有就回查 DB（`getAccountById`）。三个赛事接口（list/create/getDetail-via-...）都走这个 helper

### 3. 报名展示用最新昵称

报名时 `displayName` 是当前 nickname 快照。用户改完昵称后赛程页和报名列表还是老名字，参赛者不认识自己。

- `tournament.service.ts` `getBracket / registrationsPublic`：批量 `user.findMany` 取最新 nickname 覆盖；DB 没改，仅展示层覆盖

### 4. admin bracket 卡片配色

admin 是浅色主题，原来 bracket 卡固定深色背景（`#181c22`），文字根本看不清。

- `TournamentDetail.tsx` 新增 `BM_CARD_THEME`：按 status 分 5 套配色
  - `pending` 灰、`ready` 黄、`in_progress` 蓝、`completed` 绿、`cancelled` 暗灰
- `BracketActionModal` vs 面板同步走主题色
- bracket 不存在时按 `draft / registering / cancelled` 给不同空状态文案
