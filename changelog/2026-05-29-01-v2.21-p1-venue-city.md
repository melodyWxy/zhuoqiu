# P1 venue.city 落库 + 入库自动地理编码 + 城市筛/距离排序

## 动机
v2.21 第二阶段。P0 已经把 lat/lng → city 的服务端能力封装好了，
P1 把 city 落库到 Venue 表，并把 venue 增改流程接上正向地理编码 ——
这样新审核通过的球房自动有 city，旧的用一次性脚本回填。
列表接口同步加 `?city=&lat=&lng=` 三个查询参数，为 P2 客户端按城市筛
和同城距离排序铺路。

参考 PRD：[`prd/v2.21-nearby-by-city.md`](../prd/v2.21-nearby-by-city.md) §3.3 / §3.4 / §4 / §5。

## PRD / 设计变化
按既定 P1 范围执行，无 PRD 调整。

补一条「平台 admin 直接编辑 venue」的暂未覆盖项，已在「遗留问题」记录。

## 代码变化

### 数据库
- `prisma/schema.prisma`：`Venue` 加 `city String? @db.VarChar(32)` + `@@index([city])`
- `prisma/migrations/20260529062432_add_venue_city/migration.sql`：
  ```sql
  ALTER TABLE "venues" ADD COLUMN "city" VARCHAR(32);
  CREATE INDEX "venues_city_idx" ON "venues"("city");
  ```

### GeoService 扩展
- `server/src/geo/geo.service.ts`：
  - 新增 `resolveAddress(address) → { lat, lng, city, province }`，
    调腾讯地图 `/ws/geocoder/v1/?address=...`（正向地理编码）
  - 不缓存（每个地址一般只查一次，命中率低）
  - 失败统一返回全 null，让调用方自行决定要不要写入

### Venue 写入流程接 GeoService
- `server/src/venue/venue.module.ts`：`imports: [..., GeoModule]`
- `server/src/venue/venue.service.ts`：
  - `VenueService` 注入 `GeoService`
  - `updateOwnVenue`：当 `patch.address !== undefined` 时调用
    `geo.resolveAddress(patch.address)`，把返回的 `city/lat/lng` 写入；
    解析失败 → 三字段都置 null（地址变了，旧的 geo 一定也无效）
- `server/src/venue/venue-application.service.ts`：
  - `VenueApplicationService` 注入 `GeoService`
  - `approve` 流程重构：先在事务**外**做 `findUnique` + 状态校验 + 调
    `geo.resolveAddress(payload.address)`（避免 5s 网络等待占着行锁），
    事务内重读重检防并发，再用 geo 结果填入 `tx.venue.create()`

### 列表/详情接口
- `server/src/venue/venue.controller.ts`：`ListVenuesQueryDto` 加
  `city?: string`、`lat?: number`、`lng?: number`，转发给 service
- `server/src/venue/venue.service.ts`：
  - 抽 `VENUE_PUBLIC_SELECT` 共用 select 常量，新增 `city/lat/lng` 三字段
  - `listPublic` 当用户传 `city + lat + lng` 时取该城市全量后内存按欧氏距离平方
    排序（同城范围 < 100km 误差忽略，无需 Haversine / PostGIS）；
    没传 lat/lng → 沿用 `createdAt desc` 兜底
  - `getPublic` 复用同一 select
- `server/src/venue/tournament.service.ts`：`detailPublicById` 的
  `venue` select 加上 `city`，给 P3 跨城报名比对用

### 一次性回填
- `server/scripts/backfill-venue-city.ts`：
  - 默认 dry-run，加 `--apply` 才写库
  - 遍历 `city == null` 的 venue，调腾讯地图正向解析 → 写 `city/lat/lng`
  - 每条间 sleep 200ms 防 QPS 限流
  - 失败的打日志、不动，下次再跑可重试
- `server/package.json`：加 `"backfill:venue-city": "ts-node scripts/backfill-venue-city.ts"`

## 遗留问题 / 已知限制
- **平台 super_admin 没有直接编辑 venue 的接口**：现状只能通过商家
  `PATCH /v1/venue/me/venue` 改地址（会自动重新地理编码），或者重跑回填脚本。
  如果 P3/P4 之后发现需要平台手填 city 的兜底，再加一个 admin endpoint。
- **跨城判断口径**：腾讯地图返回的 `ad_info.city` 直辖市会是「北京市」/「上海市」
  这种省=市同名值。P3 里前端只做严格相等比对就够，不做特判。
- **腾讯地图签名 + 中文地址**：`buildSignedUrl` 目前对 query 值不做 URL-encode，
  签名串里直接带原始中文。fetch 发请求时 Node.js 会自动 percent-encode，腾讯
  服务端在校验 sig 前会 decode 回原文 —— 实测中文地址（北京朝阳区测试路 1 号）
  解析成功，方案可行。如果未来遇到包含特殊符号（`&` / `=` / `#`）的地址，
  再切到「encode 一致」的方案。
- 多实例部署时 `resolveAddress` 不共享缓存（不过它本来就不缓存，问题不大）。
