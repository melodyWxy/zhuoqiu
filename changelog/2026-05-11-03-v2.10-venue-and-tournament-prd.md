# v2.10 分支启动：球房入驻 + 赛事 PRD / 设计 / 数据模型

## 动机

v2 主干（共享比赛 + 账号 + WS + 管理后台）已稳定。接下来要切入 B 端：让球房老板能入驻、发赛事，让 C 端用户能在球房里打比赛。

按项目规则"改需求先改 PRD / 设计稿再改代码"，v2.10 分支第一步不写代码，只出设计。

## 分支

- 基于 `v2 @ 3c7c464` 切出 `v2.10`
- v2.10 专做球房与赛事模块；v2 维持稳定

## 关键决策

与用户 review 确认的方向：

1. **商家体系独立**：新建 `venue_accounts` 表，与 C 端 `users` 解耦。老板用手机号登录商家后台。理由：一店多人、一人多店的 N:M 关系放个人账号里会拧。
2. **三种赛事形态都建模**：日/周/月排行榜、报名制单场、长期联赛；v2.10 只实现**报名制单场（单败淘汰）**，其他在 schema 上预留字段。
3. **不碰钱**：报名费/奖金纯记录性字段，平台不代收代付。合规门槛和技术复杂度都压到最低。
4. **本期范围**：入驻 → 审核 → 发赛事 → 报名 → bracket 生成 → 现场开局计分 → bracket 推进 → 冠军。完整 E2E 闭环。
