# v2.20：Caddy 反向代理 + 自动 HTTPS（容器化）

让 docker-compose 一份配置就能完整起服，不再依赖宿主机预装 Caddy / Nginx。

## 改动

### `docker-compose.yml`

- 新增 `caddy` 服务（`caddy:2-alpine`）
  - 暴露 `80 / 443 / 443/udp`（HTTP/3）
  - 挂载 `./deploy/Caddyfile`
  - 持久化卷 `caddy_data`、`caddy_config`（**重要**：证书必须持久化，否则每次重启都重签会被 Let's Encrypt rate-limit）
  - depends_on: server / admin / h5
- `server / admin / h5` 的 `ports` **默认注释掉**：流量统一从 caddy 容器进，业务容器只在 docker network 内被 caddy 访问，不再对宿主暴露
- LAN 调试需要直连业务端口时手动解开

### `deploy/Caddyfile`

三个子域，每段 `reverse_proxy <docker-service-name>:port`：

- `billiards-client.macrobit.com.cn` → `h5:80`
- `billiards-admin.macrobit.com.cn` → `admin:80`
- `billiards-server.macrobit.com.cn` → `server:3001`

WebSocket Caddy 会自动透传 Upgrade/Connection；上传体积 20MB 上限。

### `.env.example` / `.env.prd`

加注释说明端口默认不暴露、需要 LAN debug 时怎么打开。

## 部署流程（最终版）

```bash
# 1. DNS：三个子域 A 记录指公网 IP（已提前做好）
# 2. .env：cp .env.prd .env
# 3. 起服：
docker compose up -d --build
# 4. 看 caddy 拿证书：
docker compose logs -f caddy
#    期望出现 "obtained certificate for billiards-...macrobit.com.cn"
```

注意：

- 80 / 443 端口必须在公网防火墙打开（Let's Encrypt HTTP-01 challenge 需要 80）
- 服务器上不要再有别的服务占用 80 / 443

## 验证

- [x] `docker compose config` 通过
- [ ] 第一次 `up -d` 后 Caddy 在 ~30 秒内拿到三张证书
- [ ] 浏览器访问三个 https 子域均 200，证书有效
- [ ] WS 心跳通过 caddy 不被中断（联机记分跑 1 分钟无重连提示）

## 不做

- HTTP→HTTPS 强制跳转：Caddy 默认就给做了，不用手动配
- 单独的 `caddy reload` 脚本：直接 `docker compose exec caddy caddy reload --config /etc/caddy/Caddyfile`
