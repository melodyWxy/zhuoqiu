# 二期 · WebSocket 实时推送 + 账号合并

## 动机

共享比赛需要实时看到对方记分；同时 C 端还需要"绑定手机号 / 跨小程序账号合并"的能力。这两块是 C 端接入前的最后两个服务端能力，本轮一起落地。

## 代码变化

### 新增

- **WebSocket Realtime Hub**（`server/src/realtime/`）
  - `realtime.service.ts`：维护 `matchId → Set<WebSocket>` 订阅表，提供 `broadcastMatchEvent` / `broadcastMatchState` / `broadcastKicked`；断开时自动清理反向索引
  - `realtime.gateway.ts`：`@WebSocketGateway({ path: '/ws' })`
    - 连接握手：`?token=&channel=user|admin`，JWT 鉴权；失败 close 4001
    - 消息：`subscribe_match`（带 `afterSeq` 拉增量重放）/ `unsubscribe_match` / `subscribe_all`（仅 admin） / `heartbeat`
    - 服务端推送：`hello` / `match_event` / `kicked` / `ack` / `error`
  - `realtime.module.ts`：`@Global` 方便任意地方注入 `RealtimeService`
- **账号合并 API**
  - `server/src/users/users.service.ts`：`bindPhone` / `unbindPhone` / `mergeUsers` / `unbindWechatById` / `unbindDouyinById`
  - `POST /v1/me/bind-phone`：验证码 + 绑定；若号已被其他账号占，返回 `{bound:false, conflictUserId}` 让客户端走合并
  - `POST /v1/me/unbind-phone`：要求还存在至少一个三方绑定，避免无法登录
  - `POST /v1/me/merge-accounts`：`{phoneNumber, code, strategy: keep_current | keep_other}`；把 secondary 的微信/抖音绑定 + matches 归属迁到 primary，secondary 标记 deleted
  - `POST /v1/admin/users/:id/unbind-wechat` / `unbind-douyin`
  - `POST /v1/admin/user-merge`（独立路径避免被 `:id` 吞掉）：super_admin 手动合并 + 审计
- **MatchService ↔ WS 集成**（`server/src/match/match.service.ts`）
  - `withMatchLock` 现在维护一个事务级 `broadcasts: Array<() => void>` 队列
  - `insertEvent` 把广播闭包 push 到队列；事务 commit 后再执行 —— **事务回滚不会泄漏幽灵事件**
  - `kickByAdmin` 额外广播 `kicked` 事件
- **main.ts**：`app.useWebSocketAdapter(new WsAdapter(app))`

## 验证步骤

### WebSocket 实时链路 (`/tmp/smoke-v2-ws.js`)

```
✓ A/B 两用户微信登录
✓ A 创建房间，B 扫码加入 slot 2
✓ B 建 WS 连接，发送 subscribe_match → 收到 ack
✓ A 发 POST /events 普胜事件 → B 通过 WS 收到 match_event (serverSeq:2)
✓ A 调 POST /end → B 收到 match_event type=end (serverSeq:3)
```

### 账号合并 (`/tmp/smoke-merge.js`)

```
✓ D 手机号登录（phone=+8613911112222）
✓ C 微信登录（无 phone）
✓ C 尝试绑定同一 phone → 返回 {bound:false, conflictUserId:D}
✓ C 调 merge-accounts keep_current → {merged:true, primary:C, secondary:D}
✓ C 的 /me 现在显示 phone 和微信 binding（D 的 phone 迁过来）
```

## 架构要点

**WebSocket 广播的事务一致性**

事务 commit 之前不广播（避免 rollback 后客户端还以为发生了）：

```typescript
private async withMatchLock(matchId, fn) {
  const broadcasts: Array<() => void> = []
  const result = await this.prisma.$transaction(async (tx) => {
    tx._broadcasts = broadcasts  // 挂在 tx 上
    await pg_advisory_xact_lock(...)
    return fn(tx)
  })
  for (const b of broadcasts) b()  // commit 成功后再广播
  return result
}
```

`insertEvent` 把广播闭包 push 到 `tx._broadcasts`，事务成功后才会执行。

**账号合并的外键迁移**

`mergeUsers` 在一个事务内：
1. `wechat_bindings`、`douyin_bindings` 的 userId 改到 primary
2. `matches.owner_user_id` + `match_players.user_id` 改到 primary
3. 如 primary 无 phone 但 secondary 有 → 先把 secondary 的 phone 置 null（释放唯一约束）再迁到 primary
4. secondary.status = deleted

## 遗留

- Refresh token rotation 仍未做
- WS 心跳服务端主动 ping 没做（只依赖客户端发心跳）
- 多实例部署时 `RealtimeService` 的订阅表是本地内存，需 Redis pub/sub 分发；MVP 单实例不影响
