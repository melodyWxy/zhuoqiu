# v2.20 起步：API 域名拆分

按"形态 2"部署：三个独立子域名

| 角色 | 域名 |
|---|---|
| C 端 H5 | `billiards-client.macrobit.com.cn` |
| 管理后台 | `billiards-admin.macrobit.com.cn` |
| 服务端 API + WS | `billiards-server.macrobit.com.cn` |

前端不再走"同源 + nginx 反代"，直接打到 API 子域。本次只动配置层，不动业务逻辑。

## 服务端

- `main.ts`：`buildCorsConfig()` 读 `CORS_ORIGINS`（逗号分隔）做严格白名单；留空 / `*` 维持原"反射任意 origin"行为，dev / 同源部署不受影响
- `docker-compose.yml`：把 `CORS_ORIGINS` 和 OSS 系列 env 透到 server 容器（之前漏了，OSS 启用没法生效）

## admin-web

- 新增 `.env.production`：`VITE_API_BASE_URL=https://billiards-server.macrobit.com.cn/v1`
- `vite build` 默认走 production mode，会自动加载这个文件 → `client.ts` / `venue-client.ts` 的 `import.meta.env.VITE_API_BASE_URL` 拿到生产域名
- `.env.example` 加注释说明拆域名时的写法

## billiards-score (Taro H5)

- `config/index.js` `defineConstants` 注入两个编译时常量，从 build 时的 `process.env` 读：
  - `TARO_APP_API_BASE`
  - `TARO_APP_WS_BASE`
- `src/core/api/config.ts` `resolveBase()` 优先读上面两个常量；都为空再回落到原"同源 / dev port 推断"逻辑
- 生产构建命令：

  ```bash
  TARO_APP_API_BASE=https://billiards-server.macrobit.com.cn/v1 \
  TARO_APP_WS_BASE=wss://billiards-server.macrobit.com.cn/ws \
  npm run build:h5
  ```

## 根 .env.example

- 三个 `*_DOMAIN` 占位换成真实域名
- 新增 `CORS_ORIGINS` 默认带上 client + admin 两个 https origin

## 部署侧需要做的

1. 三个子域名 DNS 都 A 到服务器公网 IP
2. 前面套 Caddy / Nginx 终止 HTTPS：
   - `billiards-client.macrobit.com.cn` → `host:8080`（h5 容器）
   - `billiards-admin.macrobit.com.cn` → `host:8081`（admin 容器）
   - `billiards-server.macrobit.com.cn` → `host:3001`（server 容器，必须开放对外）
3. `.env` 设置：

   ```
   CORS_ORIGINS=https://billiards-client.macrobit.com.cn,https://billiards-admin.macrobit.com.cn
   ```

4. h5 build 时带 `TARO_APP_API_BASE` / `TARO_APP_WS_BASE`；admin build 自动读 `.env.production`
