# P0 服务端地理能力 + /v1/geo/resolve-city

## 动机
v2.21「附近改按城市」第一阶段：先把服务端的地理能力封装好，
为后续 P1（venue.city 落库）和 P2（客户端定位）做底座。
本期只落逆地址解析（lat/lng → city），不动数据库 schema、不动客户端。

参考 PRD：[`prd/v2.21-nearby-by-city.md`](../prd/v2.21-nearby-by-city.md) §3.2 / §4.1。

## PRD / 设计变化
本次按既定 P0 范围执行，无 PRD 调整。

## 代码变化

### 配置
- `server/src/config/configuration.ts`：`AppConfig` 加 `tencentMap.key`，
  从 `TENCENT_MAP_KEY` 读环境变量；未配时下游会返回 `city: null`。
- `server/.env.example`：在抖音段下面加一段 `TENCENT_MAP_KEY=`，附申请地址注释。

### 新增模块 `server/src/geo/`
- `geo.service.ts`：核心逻辑
  - `resolveCity(lat, lng) → { city, province }`
  - **网格缓存**：lat/lng 按 0.01° 网格（≈ 1km）取整作 key，进程内 `Map`，
    TTL 30 天，软上限 5 万条（FIFO 淘汰），用于压腾讯地图配额。
  - **腾讯地图调用**：`GET /ws/geocoder/v1?location=lat,lng&key=...&get_poi=0`，
    5s 超时 + AbortController；任何失败（key 缺失 / 配额耗尽 / 超时 /
    `status != 0`）统一返回 `{ city: null, province: null }`，让前端降级。
- `geo.controller.ts`：`POST /v1/geo/resolve-city`，公开接口（无 token），
  body `{lat, lng}` → `{city, province}`。配额风险由网格缓存兜底，本期暂不加
  speed-limit 中间件，等观测到滥用再补。
- `dto/resolve-city.dto.ts`：`class-validator` 校验 lat ∈ [-90, 90]、
  lng ∈ [-180, 180]，配合全局 ValidationPipe 的 `transform: true` 做 number 转换。
- `geo.module.ts`：`exports: [GeoService]`，方便 P1 在 venue 模块里复用。

### 接线
- `server/src/app.module.ts`：`imports` 末尾加 `GeoModule`。

## 验证步骤
- [x] `tsc --noEmit` 通过（已跑）。
- [ ] 本地 `.env` 填好 `TENCENT_MAP_KEY` 后启服务，发请求：
      ```bash
      curl -X POST http://localhost:3001/v1/geo/resolve-city \
        -H 'Content-Type: application/json' \
        -d '{"lat":36.6512,"lng":117.1201}'
      # 预期返回 {"data":{"city":"济南市","province":"山东省"}}
      ```
- [ ] 不配 key 的情况下返回 `{ city: null, province: null }`，且日志有
      「TENCENT_MAP_KEY 未配置」warn。
- [ ] 同一 lat/lng 二次请求命中缓存（不会再有 outbound HTTP，可加
      `console.log` 临时验证或用 nock）。

## 遗留问题 / 已知限制
- 进程内缓存：多实例部署各自缓存，配额节省效果按实例数线性削弱。等真上量
  或服务上 K8s 时换 Redis。
- 没加 rate limiter：MVP 期靠网格缓存+客户端 24h 缓存联合压量。如果观测到
  异常调用峰值再补 `@nestjs/throttler`。
- 行政区划口径以腾讯地图返回为准（`ad_info.city`），直辖市返回「北京市」/
  「上海市」等，与省级同名 —— 后续比对城市名时需要逐字相等即可，不做特判。
