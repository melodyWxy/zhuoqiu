# 桌球计分 · 后台管理说明文档（平台运营视角）

> **版本：** v2.20
> **适用对象：** 平台运营 / 客服 / 数据分析（`platform_admin` 角色）
> **访问地址：** `https://billiards-admin.macrobit.com.cn`（生产）
> **技术栈：** React · Vite · admin-web

> 本文档讲**平台运营**的视角。商家自己用的后台见《球房商家端说明文档》。
> 同一份 admin-web 代码，按登录账号的 JWT type 路由到不同菜单：
> `platform_admin` → 平台菜单；`venue_account(owner/staff)` → 商家菜单。

---

## 1. 平台运营负责什么

平台运营是这个系统的"地主"，三件主线业务：

1. **入驻审核**：把控球房质量（看营业执照，过滤刷单 / 黄赌毒 / 不正规店铺）
2. **赛事 & 比赛监管**：跨球房看盘，处理纠纷（比赛作弊 / 报名退款投诉 / 商家违规）
3. **基础数据运营**：用户增长、活跃球房、活跃赛事，定期看趋势

平台**不做的事**（明确边界）：

- 不收钱、不代收代付（报名费/奖金都商家线下处理）
- 不主动给商家拉客（不是美团那种营销平台）
- 不做赛事评分 / 推荐算法（v2.x 都不做）

## 2. 登录与账号

- 默认账号 / 密码：`.env` 里的 `SEED_SUPER_ADMIN_USERNAME` / `SEED_SUPER_ADMIN_PASSWORD`
- **首次登录必须立即改密码**
- 平台管理员账号是用户名/密码登录，跟商家手机号验证码登录走的是不同账号体系
- 多个平台运营 / 客服都可建账号，目前没有细粒度 RBAC（v2.11+ 再做），所有 `platform_admin` 权限相同

## 4. 入驻审核（核心工作流）

### 4.1 看到什么

「入驻审核」列表页：所有 `state = pending` 的申请，按提交时间倒序。每行：

- 申请人手机号 + 提交时间
- 店铺名 + 地址（缩略）
- 营业执照缩略图
- [ 详情 ] 按钮

### 4.2 详情页

- 完整店铺信息：店名 / 地址 / 联系电话 / 台桌数 / 营业时间 / 介绍
- **营业执照原图**（点击放大）
- 身份证图（如果填了，本期选填）
- 历史动作：之前是不是被驳回过、什么原因
- 操作区：[ 通过 ] [ 驳回（必填原因）] [ 标记可疑（不操作但置顶）]

### 4.3 审核标准（运营手册）

**必驳回**：

- 营业执照图模糊、PS 痕迹明显、店名 / 地址与申请表对不上
- 店名含敏感词（黄赌毒、政治、医疗保健词等）
- 重复申请（同一执照号 / 同一手机号已有审核中或被封禁的店铺）

**建议驳回**：

- 信息明显不真实（比如台桌数 100 但执照是夫妻店规格）
- 营业范围与桌球台球**完全无关**（比如餐饮 / 服装类）—— 这条本期不强卡，先 review 后通过

**可通过**：

- 信息基本完整 + 执照真实 + 营业范围含娱乐 / 体育 / 桌球类

> 默认运营 1 人审核 1 单耗时 < 30 秒。月均 300 单以下时一个人足够。

### 4.4 通过后系统做什么

1. 创建 `venue` 行，绑定 `applicantAccountId` 为 owner
2. 申请状态 `pending → approved`
3. 给商家发**短信**（通过 `DEV_FIXED_SMS_CODE` 阶段先打日志，真实短信通道 v2.11+ 接）
4. 给商家发**站内信**（C 端和 admin 都能看到）
5. 写一条审计日志：`action=approve_venue_application`

### 4.5 驳回后系统做什么

1. 状态 `pending → rejected`，存 `rejectReason`
2. 商家在 admin 看到驳回提示 → 可改资料后重新提交（state 回到 `pending`）
3. 写审计日志

## 5. 球房管理

### 5.1 球房列表

跨平台所有球房，可筛：

- 状态：`active` / `suspended`
- 名称模糊搜索
- 创建时间 / 入驻方式（自助 / 后门）

每行点开是该球房的详情面板，能看：

- 店铺信息
- 当前在办的赛事
- 历史比赛量
- owner 联系方式

操作：

- [ 暂停 ]：state → `suspended`，C 端搜不到，赛事页不显示。owner 后台看到"已暂停"提示，无法发新赛事。
- [ 解除暂停 ]：恢复 `active`
- [ 编辑店铺信息 ]：平台运营手动改资料（用得少，主要给客服救场用）

### 5.2 + 新建球房（后门）

> ⚠️ 这是**后门入口**，跳过自助申请流程。所有操作走审计日志。

适用场景：

1. 合作球房快速上架（市场 BD 谈下来的）
2. 商家自助申请被驳回但来电求情，运营人工救场
3. demo / 培训用

操作流程：

1. 填完整店铺信息（与申请表一致）
2. 填老板手机号：
   - 该手机号已存在 `venue_account` → 直接绑定为 owner
   - 不存在 → 自动创建商家账号（首次登录用验证码登录）
3. 选 `state` = `active`
4. 提交 → 直接生成 venue，不走审核
5. 自动写审计日志：`action=create_venue_directly`，`adminId=当前操作人`

## 6. 赛事总览

跨球房看所有赛事，按状态筛：

- `draft`（商家草稿，未发布）
- `registering`（报名中）
- `in_progress`（进行中）
- `completed`（已结束）
- `cancelled`（已取消）

主要用途：

- **看实时盘面**：今晚有几场赛事在进行、哪场报名爆满了
- **干预纠纷**：商家强行 walkover 玩家投诉、比赛重大作弊判决
- **数据导出**：月报 / 季报需要

平台运营**不能**直接发赛事 / 编辑赛事内容（只读视角 + 强制取消能力）。

## 7. 审计日志

所有敏感操作落 `audit_log` 表，列出：

| 字段 | 说明 |
|---|---|
| timestamp | 操作时间 |
| admin_id | 操作人 |
| action | 动作枚举（approve/reject/create_venue_directly/suspend_venue/...）|
| target_type / target_id | 操作对象 |
| metadata | JSON：变更前后值、备注 |
| ip / user_agent | 来源 |

支持按时间 + admin + action 类型筛选。**审计日志不可删改**（管理员账号没有删除权限），保留至少 1 年。

## 9. 紧急情况手册（运营 SOP）

### 9.1 商家投诉用户作弊

1. 进比赛详情 → 看 `match_event` 完整事件流（每杆得分都有时间戳和操作人）
2. 看 IP / 设备 ID 是否异常（同一设备代他人加分）
3. 决策：
   - 证据充分作弊 → 标记该用户 `state=banned`，作废比赛结果
   - 不充分 → 站内信回复双方"已记录，不予判决"

### 9.2 用户投诉商家强行 walkover

1. 看赛事 bracket 历史 → 是否标记 walkover 时该对阵未到比赛时间
2. 看商家这场赛事的所有 walkover 记录 → 是否存在系统性骚操作
3. 决策：
   - 个例 → 站内信调解
   - 系统性 → 暂停该球房 + 联系 owner 沟通

### 9.3 营业执照过期 / 信息变更

v2.10 暂无主动复审机制。商家来电报备 → 后台进店铺资料页改 → 改营业执照图字段。v2.11+ 上线"年审复核"。

### 9.4 球房关店

owner 来电 → 后台球房管理 → [ 暂停 ]。3 个月无人接管自动转 `closed`（v2.12+）。

## 10. 跟商家端的菜单区别一图流

```text
登录账号类型 → 菜单可见性
├─ platform_admin（用户名密码登录）
│   ├─ 概览（全平台）
│   ├─ 入驻审核
│   ├─ 球房管理（全部球房 + 后门建房）
│   ├─ 赛事总览（跨球房）
│   ├─ 比赛 / 用户
│   └─ 审计日志
│
└─ venue_account(owner/staff)（手机号验证码登录）
    ├─ 店铺主页（自家）
    ├─ 店铺资料（仅 owner 可改）
    ├─ 赛事管理（自家）
    ├─ 现场控台
    └─ 团队（v2.11+）

同一站、同一域名、同一份代码，按 JWT type 字段路由侧边栏。
```

## 11. 已知限制 & 路线图

- ❌ 暂无细粒度 RBAC：所有 `platform_admin` 权限相同
- ❌ 暂无运营批量操作（批量审核 / 批量站内信）
- ❌ 暂无营业执照 OCR 自动识别（运营纯人眼审）
- ⏳ v2.11：营业执照年审复核、批量操作、用户分群
- ⏳ v2.12：RBAC（主管 / 客服 / 数据分析三角色拆分）
- ⏳ v3.x：营业执照 OCR + 自动初筛
