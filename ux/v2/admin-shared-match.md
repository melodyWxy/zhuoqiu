# 共享比赛 · 管理端设计稿

> **版本：** v2.0.0 草稿
> **更新时间：** 2026-05-08
> **状态：** 设计中（等用户确认）
> **范围：** 二期批次 1 —— 共享比赛 —— 管理后台 UI
> **不含：** C 端改造（客户端接共享比赛的工作，留到批次 2）、球房赛事（批次 3+）

---

## 1. 背景与目标

### 1.1 二期批次拆分

一期已经闭环（本地计分），二期要引入服务端。考虑到工作量，把二期拆成三批次：

| 批次 | 内容 | 状态 |
|------|------|:----:|
| 批次 1 | 共享比赛 · **服务端 + 管理后台** | 🔵 当前 |
| 批次 2 | 共享比赛 · C 端（小程序 / H5 接入） | ⏳ |
| 批次 3 | 球房入驻 + 赛事（服务端 + 管理端 + C 端） | ⏳ |

本文档只覆盖**批次 1 的管理后台 UI**部分。服务端 API / 数据模型在单独文档 `shared-match-backend.md` 里细化；本文档只提及依赖关系。

### 1.2 管理后台的存在意义

共享比赛是 C 端朋友局，运营通常不介入。管理后台的必要性在三件事：

1. **排查问题**——用户反馈"比分不对 / 房间卡住"时，运营能查看任意房间的实时状态和操作日志
2. **平台治理**——违规昵称、恶意行为的账号封禁
3. **运营决策支持**——活跃度、使用趋势、异常率的数据看板

### 1.3 技术栈（已确认）

- **前端**：React 18 + Ant Design Pro + Vite
- **路由**：React Router 6
- **状态**：Zustand（与 C 端保持一致）
- **API 客户端**：axios + 全局 interceptor（鉴权 / 错误码）
- **鉴权**：JWT（access 15 分钟 + refresh 7 天）

## 2. 用户角色（后台）

独立账号体系，**不复用** C 端的微信账号。三种角色：

| 角色 | 权限范围 | 典型人员 |
|------|----------|----------|
| **super_admin** | 全部功能 + 创建/管理其他后台账号 + 系统设置 | 技术负责人 / 产品负责人 |
| **operator** | 房间管理、用户管理、查看审计日志（只读）、查看系统设置（只读） | 日常运营 |
| **readonly** | 所有页面仅查看，不能执行任何写操作 | 访客 / 数据分析师 |

**账号生命周期：**

- 首次部署时通过种子脚本创建 1 个 super_admin（账号密码在环境变量里）
- 其他账号由 super_admin 在后台内创建
- 账号可停用 / 重新启用；不支持自助注册
- 密码强度要求：≥8 位，字母+数字+符号

**C 端账号关联模型（重要）：**

C 端用户可能从**微信小程序**或**抖音小程序**进入，账号系统要在后端做"同一个人"的识别：

```text
User
  id                  string         # 内部 user id（u_xxxxxx）
  phoneNumber?        string         # 手机号（绑定后才有）
  nickname            string
  avatar              string
  primarySource       'wechat' | 'douyin'   # 首次注册来源
  wechatBinding?      WechatBinding
  douyinBinding?      DouyinBinding
  createdAt
  lastActiveAt
  status              'active' | 'banned' | 'deleted'

WechatBinding
  openId              string         # 小程序级 openId
  unionId?            string         # 跨小程序/公众号一致
  bindAt
  unbound             bool

DouyinBinding
  openId              string
  unionId?            string         # 抖音也有 unionId 概念
  bindAt
  unbound             bool
```

**识别 / 合并规则：**

1. 同一手机号 → 同一 User（不同小程序登录都关联到同一个）
2. 同一微信 unionId → 同一 User（即使没绑手机号，跨小程序内复用）
3. 同一抖音 unionId → 同一 User
4. 跨平台（微信和抖音）的合并**仅在双方都绑定同一手机号时**触发；否则视为两个独立账号
5. 用户可主动"绑定手机号"来合并已有账号（要短信验证）

后台**详情页 → "账号关联" tab** 展示所有关联记录（含已解绑的历史）。运营场景：用户反馈"我两个小程序里的记录怎么不互通"——运营查看其关联状态，引导用户去 C 端绑手机号。

**封禁交互：**

```text
[ 封禁 ] → Modal:
┌──────────────────────────────────┐
│  封禁用户 张三                    │
│                                  │
│  时长：                           │
│  ○ 1 天  ● 7 天  ○ 30 天  ○ 永久│
│                                  │
│  原因（必填）：                    │
│  ┌────────────────────────────┐  │
│  │                            │  │
│  └────────────────────────────┘  │
│                                  │
│        [ 取消 ]  [ 确认封禁 ]    │
└──────────────────────────────────┘
```

封禁效果：

- 无法登录（登录接口返回 `ACCOUNT_BANNED`）
- 已在线的 session 立即失效
- 已在进行中的比赛**不强制踢出**（避免干扰他人）；下次尝试操作时提示"账号已封禁"
- 封禁到期自动解封；也可管理员手动解封

## 5. 关键交互细节

### 5.1 权限可见性

| 菜单 | super_admin | operator | readonly |
|------|:-----------:|:--------:|:--------:|
| Dashboard | ✅ | ✅ | ✅ |
| 房间列表 / 详情 | ✅ | ✅ | ✅ |
| 房间管理操作 | ✅（全部） | ✅（除修正分数） | ❌ |
| 用户列表 / 详情 | ✅ | ✅ | ✅ |
| 封禁 / 解封 | ✅ | ✅ | ❌ |
| 数据看板 | ✅ | ✅ | ✅ |
| 审计日志 | ✅（全部） | ✅（仅自己） | ✅（仅自己） |
| 系统设置 | ✅（可改） | ✅（只读） | ✅（只读） |
| 账号管理 | ✅ | ❌ | ❌ |

对 readonly 用户，所有"写"按钮要么隐藏要么灰置。

### 5.2 实时性

- Dashboard 的指标每 30 秒自动刷新
- 房间详情页的玩家分数、操作日志通过 WebSocket 实时推送（复用服务端现有的 broadcast 能力；后台用一个特殊 token 订阅全量事件）
- 其他列表页不强求实时，用户刷新或分页操作时拉新

### 5.3 错误处理

- 所有 API 错误显示在右上角 notification
- 401 → 自动跳回登录页
- 403 → 显示"无权限"并禁用该按钮
- 5xx → 显示"服务异常，稍后再试"+ 错误追踪 id，方便用户反馈给运维

## 7. 对后端能力的依赖（供服务端设计参考）

**鉴权：**

- `POST /admin/auth/login` — 账号密码登录，返回 access/refresh
- `POST /admin/auth/refresh`
- `POST /admin/auth/logout`

**房间：**

- `GET /admin/matches` — 分页 + 筛选
- `GET /admin/matches/:id` — 详情（含玩家、事件日志）
- `POST /admin/matches/:id/force-pause`
- `POST /admin/matches/:id/force-end`
- `POST /admin/matches/:id/dissolve`
- `POST /admin/matches/:id/kick` — body: userId, reason
- `POST /admin/matches/:id/correct-score` — super_admin only

**用户：**

- `GET /admin/users` — 分页 + 筛选
- `GET /admin/users/:id`
- `POST /admin/users/:id/ban` — body: duration, reason
- `POST /admin/users/:id/unban`
- `POST /admin/users/:id/reset-nickname`
- `POST /admin/users/:id/unbind-wechat` — 解绑微信（手动修复异常绑定）
- `POST /admin/users/:id/unbind-douyin` — 解绑抖音
- `POST /admin/users/merge` — body: `{ primaryUserId, secondaryUserId, reason }`，手动合并两个 User（极端场景，super_admin only）

**用户详情响应应包含：**

```json
{
  "id": "u_xxx",
  "phoneNumber": "138****5678",
  "nickname": "张三",
  "avatar": "🎱",
  "primarySource": "wechat",
  "wechatBinding": { "openId": "...", "unionId": "...", "bindAt": "..." },
  "douyinBinding": { "openId": "...", "unionId": "...", "bindAt": "..." },
  "status": "active",
  "createdAt": "...",
  "lastActiveAt": "..."
}
```

**审计：**

- `GET /admin/audit-logs` — 分页 + 筛选

**设置：**

- `GET /admin/settings`
- `PATCH /admin/settings` — super_admin only

**账号：**

- `GET /admin/accounts`
- `POST /admin/accounts`
- `PATCH /admin/accounts/:id`
- `POST /admin/accounts/:id/reset-password`
- `POST /admin/accounts/:id/activate|deactivate`

**数据看板：**

- `GET /admin/analytics/overview?range=7d`
- `GET /admin/analytics/matches?range=7d&dim=daily`
- `GET /admin/analytics/users?range=7d`

**实时：**

- `WS /admin/realtime` — 订阅全量房间事件（管理员专用频道）

所有接口响应格式统一：`{ code: 0, data: {...}, message: string, traceId: string }`。
